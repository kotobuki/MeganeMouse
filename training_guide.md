# Edge Impulse Studioでの音声モデル学習・チューニングガイド

このドキュメントは、[Edge Impulse Studio](https://studio.edgeimpulse.com/)を使って、ESP32-S3などのマイコン上で動く音声分類モデルを作るときのノウハウをまとめたものです。[MeganeMouse](README.md)に音声クリック機能を追加する過程で試行錯誤した結果、「最初から知っていればもっと早く完成させられたのに」と感じた知見を共有することを目的としています。

具体例としてはマウスボタン操作に使う音（左ボタン用と長押し用）の検知を取り上げていますが、ここで紹介している考え方の多くは、短い目的音を扱う他のプロジェクトでも応用できるはずです。

## 1. クラス設計

機械学習モデルは、学習時に見たことのない音が入力されても、「分からない」とは答えてくれません。持っている選択肢（クラス）の中から、いちばん波形が近そうなものを強引に選んでしまいます。そのため、目的の音だけを学習させると、無関係な音が入ってきたときに、どれかの目的音として誤認識される結果になりがちです。

この問題を避けるため、目的音以外の音を受け止める「ゴミ箱」として雑音用のクラスを用意し、合計4つのクラスでモデルを構築します。

- `click`（目的音・瞬間的）：左ボタンのクリック用（押す → 短時間待つ → 離す）
- `long_press`（目的音・瞬間的）：左ボタンの長押し用（押す／離すのオルタネイト動作）
- `impulsive_noise`（雑音・瞬間的）：拍手、机を叩く音、咳払いなど
- `background_noise`（雑音・背景）：人の話し声、テレビの音、エアコンの動作音など

雑音用クラスを瞬間的なものと背景的なものに分けているのは、`click`と`long_press`の目的音はいずれも瞬間的な音なので`impulsive_noise`と取り違えやすい、という傾向が実験中に見えてきたためです。それぞれのクラスで十分な量のサンプルを集めておくと、識別の切れ味がよくなります。

## 2. データ収集

モデルの性能を大きく左右するのは、アルゴリズムよりもむしろデータの質でした。特に次の3点は、最初の段階では見落としがちで、後になって「もっと早く気づいていれば」と後悔しやすいポイントです。

### 2.1. 実機のマイクで録音する

データは必ず、実際にデプロイするのと同じハードウェアで録音してください。具体的には、AtomS3Rにマイクを接続し、`SoundSampleCollector.ino`を書き込んだものを使います。

最初はPCやスマートフォンのマイクで手軽に集めたくなりますが、これがほぼ確実に落とし穴になります。マイクごとにホワイトノイズの量や周波数特性が違うのに加えて、PCやスマートフォンのマイクは内部でノイズ低減などの信号処理をかけていることが多く、実機にデプロイした途端に精度が急降下します。「学習データ上では完璧だったのに、実機では全然使い物にならない」という状況の原因の多くは、ここにありました。

### 2.2. 誤認識した音を集めて学習させる（Hard Negative Mining）

実機でテストしてみると、「話し声を`long_press`と100%の確信度で誤認識する」「拍手を`click`と100%の確信度で誤認識する」といった、自信満々な誤認識に出会うことがあります。これは、モデルが「話し声」や「拍手」という音をそもそも知らないために、手持ちのクラスの中から一番近そうなものを選んでしまっている状態です。

対策としては、誤認識が起きた環境で、実際に間違えられた音（声、拍手、環境音など）を録音し、`background_noise`や`impulsive_noise`のクラスに追加してから再学習させます。地道な作業ですが、この「間違えた音を集めて教える」ループを何度か回すのが、実用精度に到達するいちばんの近道でした。

### 2.3. 波形の位置にばらつきを持たせる（Shift samples）

実環境では、目的の音が常に録音の「ど真ん中」に来るとは限りません。連続推論時には、音が解析窓の端で見切れたり、左右に寄ったりした状態でも正しく認識できる必要があります。

Edge Impulse Studioでは、長めに録音したサンプルを短いサンプルに切り分ける「Split sample」機能を使うときに、「Shift samples」というオプションを有効にできます。このオプションをチェックしておくと、切り分けの際に目的音の位置が中央ではなく、前後にランダムにずらされた状態で保存されます。学習データに位置のばらつきを持たせることで、実環境での見切れや位置ずれにモデルが強くなります。

長めの素材を一気に切り分けるときは、このオプションを忘れずにチェックしておくのがおすすめです。

## 3. Impulse Design（前処理と特徴量抽出）

マイコンの限られたリソースで動かすモデルでは、音のどの部分をどう切り取るか（DSP＝デジタル信号処理）の設定が、モデルの性能を根本から決めます。ここでつまずいた設定を共有します。

### 3.1. ターゲット音に合わせたWindow Sizeの調整

Edge Impulse Studioの初期設定では、解析の「窓」の長さ（Window size）は`1000ms`（1秒）になっています。ただ、今回扱う目的音は50〜120ms程度の短いものなので、1秒の窓だと大半が無音になってしまい、特徴が薄まってしまいます。

試行錯誤の末、`250ms`くらいがちょうどよいという結論に落ち着きました。この長さだと、目的音が窓の約半分を占めるようになり、特徴がはっきりします。また、後述する連続推論時のスライス分割とも、数値的な相性がよくなります。

なお、このWindow sizeは、後述するセクション7の`EI_CLASSIFIER_SLICES_PER_MODEL_WINDOW`（スライス分割）で言及する「窓」と同じものを指しています。

### 3.2. Spectrogramのパラメータ

特徴量の種類としては、Edge Impulse Studioで選べるMFCCやMFEではなく、生のSpectrogramを使うのが今回の用途には合っていました。

MFCCやMFEは、低い周波数帯域に敏感な人間の聴覚特性に合わせて設計されたもので、音声認識などの「声」を扱う用途には向いています。一方、今回目的音とした無声音や拍手のような音は広い周波数帯に特徴が散らばっているため、全周波数帯域を均等な解像度で保持するSpectrogramの方が、特徴を取りこぼしにくいと感じました。

エッジデバイスでの推論時間を抑えつつ精度を維持するための設定例は以下の通りです。

- **Frame length（例：0.016秒）**：波形を切り取る「1コマ」の長さ。サンプリング周波数が16,000Hzのとき、0.016秒にすると1コマがちょうど256サンプルになります。後述するFFT lengthと一致させておくと、音の情報が途中で欠落することを防げます。
- **Frame stride（例：0.008秒）**：次のコマに進むときにずらす幅。Frame lengthの半分にしておくと、隣り合うコマが50%重なり合うことになり、数十〜100ms程度しかない音がコマの境界で取りこぼされにくくなります。
- **FFT length（例：256）**：周波数方向の細かさ。「1コマに含まれるサンプル数」以上かつ2のべき乗である必要があります。計算能力に余裕があれば512にする選択もありますが、256に抑えておくと、後段の分類モデルの計算量（推論時間）を大きく削減できます。

## 4. Classifier（分類モデル）の構築

前処理で作ったSpectrogramを「画像」と見立てて分類する、ニューラルネットワークの構造を組み立てます。

### 4.1. ネットワーク構造

Spectrogramに現れる縦線の模様（クリック音のような瞬間的な音の特徴）を見つけ出すには、画像認識でよく使われる2次元畳み込み層（2D Convolutional layer）が相性よく働きました。

Window 250ms、FFT 256に設定したときに実際に使っている構成は以下の通りです。

```text
Input layer (3,870 features)
        ↓
Reshape layer (257 columns)
        ↓
2D conv / pool layer (8 filters, kernel size 3, 1 layer)
        ↓
2D conv / pool layer (16 filters, kernel size 3, 1 layer)
        ↓
Flatten layer
        ↓
Dense layer (32 neurons)
        ↓
Dropout (rate 0.5)
        ↓
Output layer (4 classes)
```

似た音どうしを区別しきれないと感じた場合は、以下のような変更で表現力を上げることができます。ただし推論時間とメモリ使用量は増えるので、マイコンのリソースと相談しながら調整してください。

- 畳み込み層のフィルタ数を増やす（例：`8 → 16`、`16 → 32`）
- Dense layerのニューロン数を増やす、または層を追加する

## 5. 学習データとテストデータの分割

追加したノイズデータに対してモデルが本当に賢くなったかを客観的に測るため、収集したデータを学習用とテスト用に分けておきます。

1. 新しく収集したデータは、まずすべてTraining setとしてアップロードします。
2. Edge Impulse Studioの左メニューからDashboardを開きます。
3. 画面下部の「Danger zone」にある`Perform train/test split`ボタンをクリックします。
4. これにより全データが自動的に、学習用80％・テスト用20％の比率で再配分されます。
5. 再学習した後、Model testingを実行し、初見のノイズデータに対しても誤認識せずに正しく弾けているかを確認します。

Model testingで誤認識が残っていた場合は、そのサンプルをダウンロードして確認し、セクション2.2の要領でさらにデータを追加していきます。

## 6. 実機デプロイに向けた仕上げ

モデル単体の精度が上がっても、実機で連続推論を走らせると、波形の切り取られ方のタイミングによって一瞬だけ誤認識が出ることがあります。ここでは、そうした誤認識への対処方法を紹介します。

### 6.1. 連続検知（スムージング）によるスパイクの除去

たとえば手を叩いた際の「鳴り終わりの残響部分」だけがスライスに入った瞬間に、目的音と誤認識されるようなことが起きます。このような、1フレーム分だけの誤認識（スパイク）は、モデル側で潰そうとすると本来検出したい音まで潰してしまうことがあるので、推論結果を受け取る側のコードで弾くのが定石です。

具体的には、「同じクラスが複数回（例：2回）連続で閾値を超えたときに初めて、本当にその音が鳴ったと見なす」という状態保持のロジックを推論ループに入れておきます。これだけで、実用上の堅牢性が大きく向上しました。

以下は参考用の擬似コードです。`THRESHOLD`、`run_inference()`、`is_noise()`などは、実際のモデルやフレームワークに合わせて実装してください。

```cpp
// 状態保持用の変数（グローバルまたはstatic）
int last_class_id = -1;
int consecutive_count = 0;
const int REQUIRED_FRAMES = 2;  // 必要な連続検知回数（例: 2回）

void loop() {
  // 1スライス分の推論を実行し、最も確率の高いクラスIDと確率を取得
  int current_class_id = run_inference();
  float probability = get_probability(current_class_id);

  if (probability >= THRESHOLD) {
    if (current_class_id == last_class_id) {
      consecutive_count++; // 前回と同じクラスならカウントアップ
    } else {
      last_class_id = current_class_id;
      consecutive_count = 1; // 違うクラスなら1からリセット
    }

    // 設定回数に達した瞬間のみ、本当のイベントとして処理
    if (consecutive_count == REQUIRED_FRAMES) {
      if (!is_noise(current_class_id)) {
        trigger_action(current_class_id); // クリック処理等の実行
      }
    }
  } else {
    // どの音でもない（閾値以下）場合は状態を完全にリセット
    last_class_id = -1;
    consecutive_count = 0;
  }
}
```

## 7. トラブルシューティング

ここでは、作業中に実際に遭遇したエラーと、その対処方法をまとめておきます。

### DSPのパラメータを変更した直後の`ValueError`

#### エラー

Spectrogramの`FFT length`などのパラメータを変更した直後に学習を開始すると、次のようなエラーでクラッシュすることがあります。

```text
ValueError: total size of new array must be unchanged, input_shape = [4883], output_shape = [75, 65, 1]
```

#### 原因

前処理（DSP）が出力した特徴量の総数と、分類モデルの入り口にある`Reshape layer`が組み立てようとしている「画像」のマス目の数（列数）が一致していません。FFT lengthを変えると周波数ビンの数が変わり、それに応じて特徴量の総数も変わるのですが、`Reshape layer`の設定は自動では追従してくれません。

#### 解決策

Classifier画面で`Reshape layer`の設定を開き、`Columns`の値を新しい設定に合わせて修正します。画面右上のメニューから「Reset to default architecture」を選んで構造をリセットしてしまうのも手早い方法です。

`Columns`に指定する値は、DSPの出力を「時間方向のフレーム数」×「周波数方向のビン数」として見たときに、どちらを列にするかによって決まります。たとえばFFT length 512、Window 250ms、Frame stride 0.016秒の設定では、

- 周波数方向のビン数：`512 ÷ 2 + 1 = 257`
- 時間方向のフレーム数：`250ms ÷ 16ms ≒ 19`

となるので、`Columns`には`257`または`19`を指定することになります（どちらを「列」に割り当てるかは、好みとネットワーク構造の都合で選びます）。

### 実行時の`frame_length`に関するエラー

#### エラー

応答性（レスポンス）を上げるためには、1つの推論ウィンドウをより細かく分割（スライス）して、推論を高速に繰り返すのが効果的です。そのために`model_metadata.h`の`EI_CLASSIFIER_SLICES_PER_MODEL_WINDOW`を初期値から増やす（例：`4` → `8`）と、実行時に次のようなエラーが出ることがあります。

```text
ERR: frame_length (800) cannot be larger than signal's total length (500)
```

#### 原因

スライス数を増やすと、1回の処理で渡される音声波形の長さ（スライスサイズ）が短くなります。この長さが、DSPが要求する「最小フレーム長（frame length）」を下回ってしまうと、処理できずにエラーになります。

#### 解決策

DSP側で要求する最小フレーム長を下げることで、スライス数を増やしてもエラーが出なくなります。

1. Edge Impulse StudioのDSP（Spectrogram / MFE）の設定画面を開きます。
2. Parameters内の`Frame length`の値を小さくします（例：`0.05`秒 → `0.02`秒など）。
3. 特徴量を再生成し、再学習（Retrain）します。
